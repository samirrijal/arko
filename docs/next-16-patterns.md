## 1. Async `params` and `searchParams`

**Source:** `01-app/03-api-reference/03-file-conventions/page.md`

In Next 16, both `params` and `searchParams` are **promises** that must be `await`-ed (server components) or `use()`-d (client components). The Next 15 backward-compat synchronous access path is removed.

### Server component (default)

```tsx filename="apps/web/src/app/studies/[id]/page.tsx"
export default async function StudyPage(props: PageProps<'/studies/[id]'>) {
  const { id } = await props.params
  const search = await props.searchParams  // optional
  // ... fetch study, render
}
```

`PageProps<'/studies/[id]'>` is a **globally available** helper — no import. Generated by `next dev` / `next build` / `next typegen` from the route literal. Strongly types `params` keys against the actual `[bracket]` segments.

### Client component

```tsx
'use client'
import { use } from 'react'

export default function FilterBar({ searchParams }: { searchParams: Promise<{ q?: string }> }) {
  const { q } = use(searchParams)
  // ...
}
```

### Layouts

Layouts receive `params` (no `searchParams` — layouts don't re-render on search-param change):

```tsx filename="apps/web/src/app/projects/[projectId]/layout.tsx"
export default async function ProjectLayout(props: LayoutProps<'/projects/[projectId]'>) {
  const { projectId } = await props.params
  // ...
}
```

### Arko routes that need this

| Route | params shape |
|---|---|
| `/processes/[id]` | `{ id: string }` — process detail |
| `/projects/[projectId]/studies/[studyId]` | `{ projectId: string, studyId: string }` |
| `/projects/[projectId]/studies/[studyId]/calc` | same as above |
| `/processes` | `searchParams`: `{ q?: string, geo?: string, source?: string }` for filter bar |
| `/dashboard` | static, no params |

### Pattern to AVOID

```tsx
// WRONG — Next 15 sync access removed in Next 16
export default function Page({ params }: { params: { id: string } }) {
  return <div>{params.id}</div>  // type error AND runtime fail
}
```

## 2. Cache Components and `"use cache"`

**Source:** `01-app/03-api-reference/01-directives/use-cache.md`, `04-functions/cacheLife.md`, `04-functions/updateTag.md`

Caching is **opt-in and explicit** in Next 16. To enable, add to `next.config.ts`:

```ts
const nextConfig: NextConfig = {
  cacheComponents: true,
}
```

Then `'use cache'` at file/component/function level marks something cacheable.

### The three timing properties

Every cached scope has `stale` (client cache window), `revalidate` (server background regen), `expire` (max time before forced regen). Use named profiles:

| Profile | revalidate | Use case |
|---|---|---|
| `seconds` | 1s | real-time data |
| `minutes` | 1min | frequently updated |
| `hours` | 1hr | multi-daily updates |
| `days` | 1day | daily updates |
| `weeks` | 1week | weekly updates |
| `max` | 30days | rarely changes |
| `default` | 15min | when omitted |

### Pattern: cached function

```tsx
import { cacheLife, cacheTag } from 'next/cache'

export async function getProcessCatalog(databaseId: string) {
  'use cache'
  cacheLife('weeks')
  cacheTag(`processes:${databaseId}`)

  return await db.processes.findMany({ where: { databaseId } })
}
```

### Invalidation: `updateTag` vs `revalidateTag`

| | Where callable | Behavior on next request |
|---|---|---|
| `updateTag('processes')` | **Server Actions only** | Waits for fresh data (read-your-own-writes) |
| `revalidateTag('processes', 'max')` | Server Actions + Route Handlers | Serves stale, regenerates in background |
| `revalidateTag('processes')` (no profile) | same | Legacy = same as `updateTag` |

For Arko: when a user imports a new ILCD bundle, the import Server Action should call `updateTag('processes:${databaseId}')` so the Process Browser immediately reflects the new flows.

### Arko caching shape

| Surface | Cacheable? | Profile / strategy |
|---|---|---|
| Process catalog (imported bundles) | Yes | `cacheLife('weeks')` + `cacheTag('processes:${dbId}')`; `updateTag` on import |
| LCIA method metadata (5 registered presets) | Yes | `cacheLife('max')` — essentially static; only changes on engine release |
| Database license docs / metadata | Yes | `cacheLife('max')` |
| Study definition (the user's draft) | **No** | request-time; mutation-heavy |
| Calculation results | **No** | request-time; depends on study state at moment of run |
| Org/user/project metadata | **No** | request-time + RLS-gated; per-user view |
| EPD output preview | Conditional | cache the generated LCAx payload by `cacheTag('study:${id}:lcax')`, `updateTag` on study edit |

### Constraints (build-time errors if violated)

- **Can't read `cookies()`, `headers()`, `searchParams` inside `'use cache'`.** Read outside, pass in as args.
- **Args/return values must be serializable.** No class instances, no `URL`, no functions (except as pass-through).
- **`React.cache` is isolated** inside `use cache` boundaries — pass data in via args, not via outer-scope storage.

### Pattern to AVOID

```tsx
// WRONG — auth state leaks into cache key, build hangs / errors
async function getStudies() {
  'use cache'
  const session = await getSession()  // reads cookies — forbidden inside cache
  return db.studies.findMany({ where: { orgId: session.orgId } })
}

// CORRECT — pass orgId in
async function getStudies(orgId: string) {
  'use cache'
  cacheTag(`studies:${orgId}`)
  return db.studies.findMany({ where: { orgId } })
}
```

## 3. `proxy.ts` (replaces `middleware.ts`)

**Source:** `01-app/03-api-reference/03-file-conventions/proxy.md`, `01-app/02-guides/authentication.md`

`middleware.ts` is **deprecated and renamed to `proxy.ts`** in Next 16. The rename also reflects scope: this is a network-edge proxy, not Express-style middleware. Codemod available: `npx @next/codemod@canary middleware-to-proxy .`.

### File location and shape

`apps/web/src/proxy.ts` (or `apps/web/proxy.ts` if not using `src/`). Single export, named `proxy` or default:

```ts filename="apps/web/src/proxy.ts"
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export async function proxy(request: NextRequest) {
  // optimistic auth checks here — read cookie, redirect if needed
  return NextResponse.next()
}

export const config = {
  matcher: ['/((?!api|_next/static|_next/image|favicon.ico).*)'],
}
```

### Runtime change in Next 16

Proxy now defaults to **Node.js runtime** (was Edge in earlier versions). The `runtime` config option is **not available** in Proxy files — setting it throws an error. This matters for Supabase Auth: most of `@supabase/ssr` works fine on Node, no Edge constraints to worry about.

### Auth pattern (per the Next 16 authentication guide)

The doc is explicit: **proxy.ts does optimistic checks ONLY** (cookie decrypt → redirect). Real authorization happens in a Data Access Layer (DAL) close to the data:

```ts filename="apps/web/src/proxy.ts"
import { NextResponse, type NextRequest } from 'next/server'
import { decrypt } from '@/lib/session'
import { cookies } from 'next/headers'

const protectedRoutes = ['/projects', '/studies', '/processes']
const publicRoutes = ['/login', '/signup', '/']

export async function proxy(req: NextRequest) {
  const path = req.nextUrl.pathname
  const isProtected = protectedRoutes.some(p => path.startsWith(p))
  const isPublic = publicRoutes.includes(path)

  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)

  if (isProtected && !session?.userId) {
    return NextResponse.redirect(new URL('/login', req.nextUrl))
  }
  if (isPublic && session?.userId && path !== '/') {
    return NextResponse.redirect(new URL('/projects', req.nextUrl))
  }
  return NextResponse.next()
}

export const config = {
  matcher: ['/((?!api|_next/static|_next/image|.*\\.(?:png|svg|ico)$).*)'],
}
```

`(await cookies())` — note Next 16's `cookies()` is async too.

### Real authorization — `lib/dal.ts` (Data Access Layer)

```ts filename="apps/web/src/lib/dal.ts"
import 'server-only'
import { cache } from 'react'
import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'
import { decrypt } from '@/lib/session'

export const verifySession = cache(async () => {
  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)
  if (!session?.userId) redirect('/login')
  return { userId: session.userId, orgId: session.orgId, role: session.role }
})
```

Every page/server-action that touches data calls `verifySession()` first. RLS in Postgres (`auth.uid()` per `D-0025`) is the second line of defense.

### Layout caveat (load-bearing for Arko)

> Due to Partial Rendering, **don't put auth checks in Layouts** — they don't re-render on navigation. Do checks close to the data source.

For Arko: don't gate the projects/studies layout on `verifySession()` — let the page-level fetches do it. Otherwise users who land on a stale tab won't get redirected when their session expires.

### Server Actions and Route Handlers

`'use server'` actions and `app/api/*/route.ts` handlers are NOT covered by `proxy.ts` matcher logic for security purposes — they need their own `verifySession()` call. The proxy doc explicitly warns: "A matcher change or a refactor that moves a Server Function to a different route can silently remove Proxy coverage. Always verify authentication and authorization inside each Server Function rather than relying on Proxy alone."

## 4. Tailwind v4 `@theme` config

**Source:** Installed Tailwind v4.2.4 + `apps/web/src/app/globals.css` scaffold

Tailwind v4 ships **CSS-first config** — no `tailwind.config.ts` file by default. All theme config goes in `globals.css` via the `@theme` directive. Content paths are auto-detected; no manual `content: [...]` array.

### The default `globals.css` shape

```css filename="apps/web/src/app/globals.css"
@import "tailwindcss";

:root {
  --background: #ffffff;
  --foreground: #171717;
}

@theme inline {
  --color-background: var(--background);
  --color-foreground: var(--foreground);
  --font-sans: var(--font-geist-sans);
  --font-mono: var(--font-geist-mono);
}
```

`--color-background` becomes available as `bg-background`, `text-background`, `border-background` etc. Same naming pattern for fonts (`font-sans`), spacing, etc.

### D-0028 token translation strategy

Per [D-0028](../DECISIONS.md#d-0028), Arko inherits KarbonGarbi's token values. KarbonGarbi is on Tailwind v3 with a JS config — translation, not copy:

```css filename="apps/web/src/app/globals.css"
@import "tailwindcss";

@theme {
  /* Inherited from KarbonGarbi (D-0028) */
  --color-primary: #16A34A;
  --color-primary-foreground: #ffffff;
  --color-border: #f3f4f6;        /* gray-100 */
  --color-muted: #f9fafb;         /* gray-50 — used for D-0028 alternating rows */

  /* Arko-specific divergence (D-0028) */
  --radius-button: 0.375rem;      /* rounded-md, NOT rounded-full */

  /* Density tokens (D-0028 — practitioner register) */
  --spacing-row: 0.5rem;          /* tighter than KarbonGarbi default */
  --line-height-table: 1.25;
}

/* Dark mode if enabled (Phase 2 ships light only per D-0028 open items) */
@media (prefers-color-scheme: dark) {
  :root {
    --color-background: #0a0a0a;
    --color-foreground: #ededed;
  }
}
```

### Custom utility classes (when @theme isn't enough)

Tailwind v4 supports `@utility` for custom utilities:

```css
@utility table-striped {
  tbody tr:nth-child(odd) {
    background-color: var(--color-muted);
  }
}
```

Then use `<table className="table-striped">` for D-0028's "alternating rows in process lists" pattern.

### Pattern to AVOID

```ts
// WRONG — Tailwind v3 JS config doesn't apply in v4
// Don't create apps/web/tailwind.config.ts
import type { Config } from 'tailwindcss'
export default { content: ['./src/**/*.tsx'], theme: { extend: { colors: { primary: '#16A34A' } } } } satisfies Config
```

The v4 way is `@theme` in CSS. There's a v3-compat path documented at `node_modules/next/dist/docs/01-app/02-guides/tailwind-v3-css.md` if needed for migration scenarios, but greenfield Arko should be v4-native.
