## Abstract

This specification defines the **mathematical contract** of the Arko
calculation engine: the inputs it accepts, the outputs it produces, the
numerical guarantees it makes, and the error conditions it raises. It is
language- and implementation-agnostic. Any implementation that conforms to
this specification and passes the conformance test vectors (§14) is a
**conforming Arko calculation engine**.

This document does **not** specify performance, on-disk formats, network
protocols, or user interface. Those are covered in `specs/api/`,
`specs/sync/`, `specs/io/`, and the engine implementation's own design docs.

---

## 1. Status of this document

This is a working draft published for public review. The v1.0 specification
will be published only after:

1. The conformance test suite (§14) contains ≥10,000 studies.
2. The reference implementation passes parity tests against at least two
   independent LCA engines (Brightway 2.5 and OpenLCA 2.x) on those studies
   within the tolerances defined in §8.
3. At least one **independent** implementation of this specification exists
   and passes the conformance suite.

Until then, breaking changes may be introduced in any minor version.

## 2. Conventions

- **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, **MAY** are used per
  [RFC 2119] and [RFC 8174].
- All identifiers in code fragments are case-sensitive.
- All numbers are **IEEE 754 binary64** (`f64`) unless explicitly stated.
- All matrices are **sparse** unless explicitly stated dense.
- All vectors are column vectors.
- Indices are **0-based**.
- A "study" refers to a complete calculable LCA dataset; see §4.1.

---

## 3. Scope

### 3.1 In scope

- Static inventory analysis (matrix construction + linear solve).
- Impact assessment (characterization only; no normalization or weighting
  in v0.1 — deferred to v0.3).
- Multi-functional process handling via allocation (§6.4) and system
  expansion (§6.5).
- Parameterized inventories (scalar parameters with dependency DAG; see §5).
- Uncertainty propagation via Monte Carlo (§9).
- Incremental recalculation via Sherman-Morrison-Woodbury updates (§10).

### 3.2 Out of scope (v0.1)

- Dynamic / time-dependent inventory (deferred to v0.4).
- Regionalized impact assessment (deferred to v0.3).
- Endpoint / damage-level impact (deferred to v0.3).
- Social LCA indicators (deferred indefinitely).
- Input-output hybrid analysis (deferred to v0.5).
- Consequential modeling extensions beyond what allocation choice expresses.

## 4. Data model

### 4.1 Study

A **study** is the unit of calculation. It is the tuple:

```
Study = (
    technosphere:      SparseMatrix<f64>,   // A, n × n
    biosphere:         SparseMatrix<f64>,   // B, m × n
    characterization:  SparseMatrix<f64>,   // C, k × m
    functional_unit:   SparseVector<f64>,   // f, length n
    processes:         Vec<ProcessMeta>,    // length n
    flows:             Vec<FlowMeta>,       // length m
    impacts:           Vec<ImpactMeta>,     // length k
    parameters:        Vec<Parameter>,
    license_tiers:     Vec<LicenseTier>,    // per-process; see §11
    method_id:         String,
    method_version:    String,
)
```

where:

- `n` is the number of **technosphere processes** (activities).
- `m` is the number of **elementary flows** (substances / resources).
- `k` is the number of **impact categories** evaluated by the chosen method.

### 4.2 Matrix semantics

- **A (technosphere, `n × n`).** Column `j` of A is the **unit process
  profile** of process `j`: the amount of each technosphere product
  consumed (negative) or produced (positive) per unit of process activity.
  Convention: **diagonal dominance with positive reference product**. If
  process `j` produces its reference product at amount `d_j > 0`, then
  `A[j, j] = d_j` and all other non-zero entries in column `j` are
  consumption (negative) or co-production (positive).

- **B (biosphere, `m × n`).** Column `j` of B is the elementary flow vector
  of process `j`: emissions (positive) and resource uptake (negative, by
  convention — or positive with separate sign convention per flow; see
  §4.4) per unit of process activity.

- **C (characterization, `k × m`).** Row `i` of C is the characterization
  factor vector for impact category `i`: CF per elementary flow. A zero
  means the flow does not contribute to that category.

- **f (functional unit, length `n`).** The demand vector. Non-zero entries
  represent final demand on processes (usually a single `1.0` on the
  reference product of the studied process).

### 4.3 The three-equation pipeline

The core calculation is:

```
  A · s = f                 (scaling factors)       [Eq. 1]
  g     = B · s             (inventory result)      [Eq. 2]
  h     = C · g             (impact result)         [Eq. 3]
```

- `s` is the **scaling vector** (length `n`): how much each process must
  run to satisfy demand `f`.
- `g` is the **life-cycle inventory** (length `m`): total elementary flows.
- `h` is the **impact score vector** (length `k`).

A conforming engine **MUST** compute `s`, `g`, and `h` such that, for the
conformance test vectors (§14), each component is within the tolerance
bounds of §8 against the reference values.

### 4.4 Sign conventions

Two conventions are permitted; a study **MUST** declare which it uses in
its `sign_convention` field:

- `"producer-positive"` (default; ecoinvent-style): products produced are
  positive, consumed are negative. Emissions to environment are positive.
- `"consumer-positive"` (ILCD-style): products consumed are positive. This
  requires a sign flip on column construction.

An engine **MUST** accept either; internally it **MAY** normalize to one.

### 4.5 Units

Every flow (technosphere row/column, biosphere row) carries a **unit**.
Units **MUST** be represented as an ISO-80000 / UCUM-compatible string
(e.g., `"kg"`, `"MJ"`, `"m3"`, `"kg.km"`, `"person.a"`).

The engine **MUST** refuse to construct A, B, or C if:

- A column of A contains entries with inconsistent units relative to the
  process's declared reference product unit, **and** the study has not
  supplied explicit unit-conversion factors.
- A row of B references a flow whose unit is incompatible with the
  characterization unit expected by any active impact category.

Unit propagation is checked at **construction time**, not solve time.
Solve-time unit errors are a specification violation.

## 5. Parameters

A study **MAY** define named scalar parameters, which appear in matrix
entries as expressions rather than literals.

### 5.1 Parameter model

```
Parameter = (
    id:          String,
    expression:  String,           // free variable OR expression over other params
    value:       Option<f64>,      // required if expression is a free variable
    unit:        String,
    dependencies: Vec<String>,     // resolved dependency ids
)
```

Expressions use a **restricted arithmetic language**:

- Operators: `+`, `-`, `*`, `/`, `^` (integer exponent only).
- Functions: `min`, `max`, `abs`, `sqrt`, `log`, `exp`, `if(cond, a, b)`.
- No I/O, no loops, no user-defined functions.
- Variables resolve to other parameters by id.

### 5.2 Evaluation

Parameters are evaluated in **topological order** of their dependency DAG.
Cycles **MUST** be rejected at construction with `E_PARAM_CYCLE`.

A matrix entry may reference at most **one** parameter expression
multiplied by a constant literal (v0.1 restriction; lifted in v0.2).

### 5.3 Partial derivatives

For uncertainty propagation (§9) and incremental recalc (§10), the engine
**MUST** be able to compute `∂A[i,j] / ∂p` for any parameter `p`. Since
entries are `constant × param_expression`, this is mechanical AD on the
expression tree.

## 6. Construction

### 6.1 Matrix assembly order

The engine **MUST** perform the following checks in this order, producing
the first-encountered error:

1. Parameter DAG well-formed (no cycles, all deps resolvable).
2. Parameter evaluation (all expressions produce finite f64; no NaN, no ±∞).
3. Unit consistency (per §4.5).
4. License-tier compatibility of the graph reachable from `f`'s support
   (per §11).
5. Matrix shape consistency (A square, B and C dims match).
6. Numerical well-conditioning heuristic (§6.2).

### 6.2 Well-conditioning heuristic

Before invoking the solver, the engine **SHOULD** check:

- A has no zero diagonal entries (after sign normalization).
- `cond_1(A)` estimated (via Higham's method or equivalent) is `< 1e12`.

If the condition number estimate exceeds `1e12`, the engine **MUST** still
attempt the solve but **MUST** attach warning `W_ILL_CONDITIONED` to the
result with the estimate value.

### 6.3 Solver requirements

A conforming engine **MUST** use a **direct sparse solver** for the primary
solve. LU with partial pivoting is the reference; Cholesky is acceptable
only for provably symmetric-positive-definite A (rare in LCA).

Iterative solvers (GMRES, BiCGStab) **MAY** be used as fallback or for very
large systems but **MUST** declare convergence tolerance in the result.

The reference implementation uses:

- Small (n < 100): dense LU via `nalgebra` or equivalent.
- Medium (100 ≤ n < 10,000): sparse LU via `faer`.
- Large (n ≥ 10,000): sparse LU via UMFPACK through Rust bindings.

This tiering is a **performance recommendation**, not a conformance
requirement. All tiers **MUST** produce results within the §8 tolerances.

### 6.4 Allocation

Multi-functional processes (processes with more than one positive
co-product in column `j` of A) **MUST** be resolved **before** solving.
The engine supports three allocation modes, selected per-process:

| Mode         | Rule                                                                       |
| ------------ | -------------------------------------------------------------------------- |
| `mass`       | Allocate by relative mass of co-products.                                  |
| `economic`   | Allocate by relative market value (requires `price` field per co-product). |
| `energy`     | Allocate by relative energy content (requires `lhv` per co-product).       |
| `user`       | User-provided allocation factors summing to 1.0 ± 1e-9.                    |
| `substitution` | System expansion; convert to a separate process subtraction (§6.5).      |

The allocation choice **MUST** be recorded in `ProcessMeta.allocation`.

### 6.5 System expansion / substitution

Substitution is handled by adding **negative** technosphere columns: the
avoided burden process is subtracted from the allocated system. The
engine **MUST** flag substitution-bearing columns in result provenance
(§12).

## 7. Numerical determinism

### 7.1 The determinism contract

Given:

- The same study bytes (canonically serialized per `specs/io/v0.1`).
- The same engine version.
- The same architecture (x86_64 or aarch64).
- The same OS family (Linux, macOS, Windows).

The engine **MUST** produce **bit-identical** results for:

- `s`, `g`, `h` vectors.
- Sensitivity matrices (§9.3).
- Uncertainty Monte Carlo samples *given the same RNG seed*.

### 7.2 Techniques for determinism

- Matrix assembly **MUST** iterate inputs in a stable, content-addressed
  order (e.g., sorted by process id hash), not by hash-map iteration order.
- Floating-point reductions (dot products, matrix-vector products) **MUST**
  use a fixed summation order. Parallel reductions **MUST** use pairwise /
  tree reduction with deterministic partitioning.
- Sparse matrix factorization **MUST** use a deterministic fill-reducing
  ordering (COLAMD with fixed tie-breaking or equivalent).
- BLAS/LAPACK calls **MUST** use a deterministic backend. Multi-threaded
  OpenBLAS is not deterministic; use a single-threaded or determinism-
  certified build.

### 7.3 Cross-architecture determinism

Cross-architecture determinism (x86_64 vs aarch64) is **NOT** guaranteed
in v0.1 due to differences in FMA rounding, `libm` trig implementations,
and subnormal handling. Results across architectures **SHOULD** agree to
within `1e-12` relative error.

Cross-architecture bit-determinism is a **v0.2 goal**.

## 8. Tolerance definitions

### 8.1 Reference parity

For a study `S` with reference result `h_ref` computed by a designated
reference implementation (v0.1: Brightway 2.5 at pinned commit), a
conforming engine's result `h` **MUST** satisfy, for every component `i`:

```
  |h[i] - h_ref[i]| ≤ max(ε_abs, ε_rel * |h_ref[i]|)
```

with:

- `ε_abs = 1e-12`
- `ε_rel = 1e-9`

### 8.2 Cross-implementation parity

For cross-parity against OpenLCA 2.x, tolerances are loosened:

- `ε_abs = 1e-9`
- `ε_rel = 1e-6`

because OpenLCA uses different characterization factor precision in its
packaged methods. The engine **MUST NOT** pass this by tuning the engine;
divergences beyond tolerance are investigated as potential bugs in either
implementation.

### 8.3 Conditioning-adjusted tolerance

For studies where `cond_1(A) > 1e8`, tolerances are multiplied by
`cond_1(A) / 1e8`. This is a concession to numerical reality, not a
weakening of the specification.

## 9. Uncertainty

### 9.1 Distribution model

A matrix entry **MAY** declare an uncertainty distribution. Supported
distributions (v0.1):

| Distribution | Parameters                         | Notes                      |
| ------------ | ---------------------------------- | -------------------------- |
| `lognormal`  | `geometric_mean`, `geometric_sd`   | ecoinvent default          |
| `normal`     | `mean`, `sd`                       |                            |
| `triangular` | `min`, `mode`, `max`               |                            |
| `uniform`    | `min`, `max`                       |                            |
| `pert`       | `min`, `mode`, `max`, `lambda=4.0` |                            |

Point values are lognormal with `geometric_sd = 1.0`.

### 9.2 Monte Carlo

- **Sampling:** low-discrepancy (Sobol') **SHOULD** be used by default;
  pseudo-random (Mersenne-Twister seeded) **MAY** be selected.
- **Seed contract:** results with the same seed + same engine version **MUST**
  be bit-identical (per §7).
- **Default iterations:** 1,000. User-configurable.
- **Correlation:** v0.1 treats all uncertain entries as independent. Pedigree
  correlations are deferred to v0.2.

### 9.3 Sensitivity analysis

For each impact category `i` and each parameter `p`, the engine **MUST**
compute:

```
  ∂h[i] / ∂p = C[i, :] · ∂g/∂p
             = C[i, :] · B · ∂s/∂p
             = -C[i, :] · B · A⁻¹ · (∂A/∂p) · s
```

This is one solve per parameter (using the already-factored A⁻¹), yielding
`O(k · n_params)` cost. For studies with many parameters, the
adjoint method **SHOULD** be used: one additional solve `A^T · y = C^T`
yields all sensitivities at `O(k + n_params)` cost. Engines **MAY** choose.

## 10. Incremental recalculation

A common user workflow is: edit one parameter, see the result update.
Refactoring A from scratch per edit is wasteful.

### 10.1 Sherman-Morrison-Woodbury update

For a rank-`r` modification `ΔA = U · V^T` (where `r` is small; typically
1-3), the updated solution is:

```
  s_new = s - A⁻¹ · U · (I + V^T · A⁻¹ · U)⁻¹ · V^T · s
```

which requires only `r` solves against the already-factored A, not a
refactorization. An engine **MUST** support this path for:

- Single parameter edit (r=1).
- Process substitution (replacing one column; r=1).
- Adding/removing an edge (r=2 in general).

Large structural edits (adding or removing processes) **MUST** trigger a
full refactorization.

### 10.2 Staleness

The engine **MUST** track a "staleness generation" counter per study. Any
incremental update increments it. Full refactorizations reset to a fresh
generation and record the reset in provenance.

## 11. License-tier propagation

### 11.1 Tier model

Every process in A carries a **license tier**:

```
LicenseTier = {
    source:            String,       // "ecoinvent-3.11", "custom-user", ...
    allow_publish:     bool,         // may impact result be published externally?
    allow_share:       bool,         // may inventory be shared across workspaces?
    allow_export:      bool,         // may raw data be exported?
    derivative_rules:  DerivativeRules,  // see §11.3
    expiry:            Option<DateTime>,
}
```

### 11.2 Effective restriction of a result

The effective restriction on a result `h` is the **join** of the
restrictions of every process `j` for which `s[j] > ε_presence`, where
`ε_presence = 1e-9`.

In plain English: **if your answer depends non-trivially on a restricted
process, your answer inherits that restriction.** The engine **MUST**
compute and attach the effective restriction to every result.

### 11.3 Derivative rules

ecoinvent and similar databases restrict **derivative** data. Arko
formalizes this as a small policy language:

```
DerivativeRule = {
    when:   { scaling_ge: 0.01 },     // trigger condition
    action: "block" | "warn" | "watermark",
    message: String,
}
```

Rules evaluate at result-publish time, not solve time. The solve always
succeeds; publishing is what gets blocked.

### 11.4 Provenance tie-in

The license-tier graph walk **MUST** be recorded in result provenance
(§12) as the list of processes that contributed to any restriction.

## 12. Provenance

Every result **MUST** carry a provenance record:

```
Provenance = {
    engine_version:     String,        // semver + git SHA
    study_hash:         String,        // blake3 of canonicalized study
    method_id:          String,
    method_version:     String,
    solver_used:        String,        // "umfpack", "faer", "nalgebra-dense", ...
    solver_config:      JsonValue,
    generation:         u64,           // staleness counter
    computed_at:        DateTime<Utc>,
    computed_by:        UserId,
    restrictions:       EffectiveRestriction,
    contributing_processes: Vec<ProcessId>,
    warnings:           Vec<Warning>,
}
```

Results without provenance **MUST NOT** be exportable or publishable.

## 13. Errors and warnings

### 13.1 Error taxonomy

Errors are fatal; the engine does not produce a result.

| Code                    | Meaning                                                        |
| ----------------------- | -------------------------------------------------------------- |
| `E_SHAPE_MISMATCH`      | Matrix dimensions inconsistent.                                |
| `E_PARAM_CYCLE`         | Parameter dependency DAG contains a cycle.                     |
| `E_PARAM_UNRESOLVED`    | Parameter references undefined variable.                       |
| `E_PARAM_NONFINITE`     | Parameter evaluation produced NaN or ±∞.                       |
| `E_UNIT_INCOMPATIBLE`   | Units incompatible at construction.                            |
| `E_SINGULAR`            | A is exactly singular; no solution exists.                     |
| `E_LICENSE_VIOLATION`   | Study references processes whose license prohibits calculation. |
| `E_METHOD_UNKNOWN`      | `method_id` not registered.                                    |
| `E_ALLOCATION_INVALID`  | Allocation factors do not sum to 1.0 (within 1e-9).            |
| `E_INTERNAL`            | Engine bug. Report to maintainers.                             |

### 13.2 Warning taxonomy

Warnings accompany a result; they do not prevent it.

| Code                    | Meaning                                                        |
| ----------------------- | -------------------------------------------------------------- |
| `W_ILL_CONDITIONED`     | `cond_1(A) > 1e8`. Result has degraded precision.              |
| `W_NEAR_SINGULAR`       | Pivot below tolerance during factorization.                    |
| `W_SUBSTITUTION_USED`   | Result includes system-expansion subtraction.                  |
| `W_MC_NONCONVERGENT`    | Monte Carlo standard error above `5e-2` after all iterations.  |
| `W_RESTRICTED_RESULT`   | Result depends on processes with publish restrictions.         |
| `W_OUTDATED_METHOD`     | Impact method has a newer version available.                   |

## 14. Conformance

### 14.1 Test vectors

The conformance suite lives at `engine/differential/vectors/` and
comprises, at v1.0 ratification:

- ≥10,000 published studies drawn from:
  - ecoinvent 3.11 (sample provided in test harness; full DB requires license)
  - ELCD reference dataset (public)
  - OpenLCA Nexus public packs
  - EF 3.1 reference studies
  - Synthetic property-test studies (§14.3)

Each vector is a triple `(study, method, expected_h)` where `expected_h`
is the result from the **consensus** of ≥2 reference implementations,
with any divergence investigated and annotated.

### 14.2 Conformance levels

| Level      | Requirement                                                      |
| ---------- | ---------------------------------------------------------------- |
| `L1-basic` | Static inventory + characterization on all L1 vectors (≥1,000).  |
| `L2-full`  | L1 + allocation + parameters on all L2 vectors (≥5,000).         |
| `L3-elite` | L2 + uncertainty + incremental + determinism on all vectors.     |

An implementation **MUST** declare its highest-passed level in its README.

### 14.3 Property-based tests

The suite also contains generative tests that any conforming engine must
pass:

- **Scaling identity:** for any study, if we scale the functional unit by
  `α`, results scale by `α`.
- **Idempotent recompute:** `solve(solve(study).provenance.study_hash)` is
  equal to `solve(study)` for byte-identical study hash.
- **Block-diagonal independence:** if A is block-diagonal, results on each
  block **MUST** equal the results computed on that block in isolation.
- **Parameter rewrite:** replacing a parameter reference with its resolved
  value **MUST** not change any result beyond tolerance.
- **Sherman-Morrison parity:** results from an incremental update **MUST**
  match full-refactorization results within §8 tolerances.

### 14.4 Reporting

A conformance run **MUST** emit a `conformance-report.json` containing,
per vector: pass/fail, tolerance used, and observed deviation. Results
**MUST** be reproducible by any reviewer given the vector set and engine
binary.

## 15. Versioning

- **Major (vX):** breaking changes to the math contract, data model, or
  conformance suite.
- **Minor (v0.X):** additions (new distributions, new error codes) that
  do not invalidate existing passing implementations.
- **Patch (v0.X.Y):** editorial changes only.

Until v1.0, any minor version may be breaking.

Implementations **MUST** declare the spec version they target.

## 16. Appendix A — Worked example

A minimal 2-process study:

```
Processes: p0 (steel, kg), p1 (electricity, MJ)
Flows:     f0 (CO2, kg)
Impacts:   GWP (kg CO2-eq)

A = [[1.0, -0.5],        # 1 kg steel needs 0.5 MJ electricity
     [0.0,  1.0]]
B = [[2.0,  0.1]]        # 2 kg CO2 per kg steel; 0.1 kg CO2 per MJ electricity
C = [[1.0]]              # GWP is 1 kg CO2-eq per kg CO2
f = [1.0, 0.0]           # demand: 1 kg steel

Solve A·s = f:
  s = [1.0, 0.5]

g = B·s = [2.0 · 1.0 + 0.1 · 0.5] = [2.05]
h = C·g = [2.05]

Result: 2.05 kg CO2-eq per kg steel.
```

This is the smallest non-trivial test vector and **MUST** be in the L1
conformance set as `vec_0000_minimal_two_process`.

## 17. Open questions for v0.2

- **Dynamic inventory.** How does time enter the model? Static A(t)
  snapshots? Continuous integration? The literature is divided.
- **Pedigree-based correlation.** How do we express "these two entries
  share an uncertainty source" without exploding the schema?
- **Regionalized impact.** Is a region tag on flows sufficient, or do we
  need a full spatial join against regionalized CFs?
- **Consequential modeling.** Is marginal vs average a method concern,
  a matrix-construction concern, or both?
- **Hybrid I-O/process.** Can IO tables be admitted as additional A blocks,
  or do they need a separate pipeline?

These are flagged now so implementers do not paint themselves into a corner.
